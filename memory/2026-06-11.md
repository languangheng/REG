## 自动分析 UI 实时日志反馈 (08:52 CST)

### Problem
点击"开始分析"后无任何 UI 反馈。原因是 `/api/analyze` SSE 端点在后台线程跑完 `auto_config` 后才一次性推送所有日志，等待期间只发心跳。

### Solution
1. **`auto_config.py`**: 加 `on_log: Optional[callable]` 参数，monkey-patch `result.log.append` 使每条日志实时触发回调
2. **`site_manager.py` `/api/analyze`**: 传 `on_log` 回调 → `queue.Queue()` → SSE 实时推送，改为边跑边推
3. 前端无需修改（已用 `EventSource.onmessage` 实时渲染）

### Key pattern
- 用 `queue.Queue` + sentinel (`None`) 实现线程间日志传递
- `on_log` 回调在 `auto_config` 工作线程中调用，`generate()` 在 Flask 线程中消费队列

## _discover_category_links href 为空 bug (10:37 CST)

### 根因
`auto_config.py` 的 `_discover_category_links()` 用 `snap.elements` 取 `<a>` 标签的 `href`，但 CDP snapshot 对 mimimi.top 这类 SPA/JS渲染站不捕获动态 href——所有 `<a>` 的 `href=""`。`snap.links()` 同样返回空 href。

### 已确认
- `snap.elements` 有 59 个元素，其中 38 个 `is_link=True`，但 `href` 全空
- `client.evaluate_js()` 可用（`browser_act.py` 已有此方法）

### 修复方案（未完成，下次继续）
- 用 `client.evaluate_js()` 执行 JS 实时提取 `<a>` 标签的 href
- 在 `BrowserActClient` 新增 `get_all_links()` 方法，用 JS `document.querySelectorAll('a')` 获取完整 href
- `_discover_category_links()` 改用此方法替代 `snap.elements`

### 当前状态
- `debug_discover.py` 已写好但因 href 全空返回 0 结果
- 修复未写入 `auto_config.py`，下次继续

## auto_config_multi 首页误判为列表页 (10:45)

### 根因
`auto_config_multi` 对首页 URL（`https://www.mimimi.top/`）调用 `_quick_check_pagination`，首页 markdown 里有大量 `/voddetail/{id}.html` 详情链接，`_cluster_links` 提取出有效模式，导致首页被**误判为列表页**，只分析了"当前页"1个入口，跳过分类发现。

### 确认结果
端到端测试输出：`entry_points count: 1`，只有一个 `name=www.mimimi.top/, type=video` 的入口。

- 前端添加「首页模式（自动发现分类）」checkbox，默认勾选
- 后端 `auto_config_multi` 新增 `is_homepage` 参数
- is_homepage=True → 直接分类发现（跳过 `_quick_check_pagination`）
- is_homepage=False → 走原有列表页判断逻辑
- 改动文件：`templates/index.html`、`site_manager.py`、`auto_config.py`
