# 项目强制工作流

> 本文件为项目级强制约束，所有任务必须遵循以下流程，无例外。

## 核心流程：Plan → 审批 → 实现 → 验证 → 提交

每个功能/任务必须按以下步骤逐一执行，**禁止跳过任何步骤**：

### 1. Plan（计划）
- 实现前必须先给出明确的 Plan
- Plan 内容包括：修改哪些文件、新增哪些方法、具体代码变更
- Plan 必须等待用户审批同意后才能动手

### 2. Implement（实现）
- 用户同意 Plan 后才能开始编码
- 一次只做一个功能，禁止同时实现多个功能
- 实现必须严格按 Plan 执行，不得擅自增加额外改动

### 3. Verify（验证）
- 实现完成后，先自动验证功能是否正常
- 自动验证通过后，交由用户手动验证
- 用户验证通过才算功能验收

### 4. Commit（提交）
- **只有用户验收通过后**，才能 git commit 当前功能的所有改动
- commit message 格式：`feat(<模块>): add <功能名>` 或 `fix(<模块>): <修复描述>`
- 每个功能独立提交，确保可独立回滚

## Git 版本管理规则

- 每个功能对应一个 commit，commit 之间不混合
- 如需回滚某个功能：`git revert <commit-hash>`
- 禁止 `git push --force`
- 禁止修改已提交的 commit（不用 `--amend`）

## 测试与临时文件规范

- 所有验证/测试脚本统一放在 `tmp/` 目录下
- 归档的旧文件统一放在 `tmp/_archive/` 目录下
- `tmp/` 已加入 `.gitignore`，不会被提交到仓库
- 禁止在项目根目录创建 `debug_*.py`、`fix_*.py`、`test_*.py`、`verify_*.py` 等临时文件
- 验证完毕后，测试脚本可保留在 `tmp/` 中作为参考，也可清理

## 禁止事项

- ❌ 未经 Plan 审批就动手写代码
- ❌ 同时实现多个功能
- ❌ 用户未验收就提交 commit
- ❌ 一次 commit 包含多个功能
- ❌ 擅自修改与当前功能无关的代码
- ❌ 在项目根目录创建临时测试/调试文件
